什么是组件？

组件 (Component) 是 Vue.js 最强大的功能之一。组件可以扩展 HTML 元素，封装可重用的代码。在较高层面上，组件是自定义元素，Vue.js 的编译器为它添加特殊功能。在有些情况下，组件也可以表现为用 is 特性进行了扩展的原生 HTML 元素。

以上摘自Vue的官网关于组件的介绍。我们的Vue应用是有多个组件构成，组件之间需要有数据传递才能让我们的应用正常运转。

数据传递

父子组件

什么是父子组件？父子组件代表的是组件之间的关系：组件A在它的模板中使用了组件B，这种情况下，组件A就是组件B的父组件，组件B是组件A的子组件。它们之间通信，父组件可能给子组件下发数据，子组件则将其内部要发生的事情告诉父组件。在父子组件下，父组件通过prop给子组件下发数据，子组件通过事件给父组件发送消息。

使用prop传递数据

组件实例作用域是孤立的，所以不能在子组件的模板里直接引用父组件的数据。父组件可以通过prop将数据传递给子组件。

前端的世界里，我们能找到各种功能的js文件。这些js文件在自己需要完成的功能上都做的很好。典型的一些插件如:

  • js.cookie.js：封装cookie操作的js

而对于其他的js库，如axios，如果要以插件的方式引入，那么我们需要做一次小小转换——将axios制作成Vue的插件。

原理

Object.defineProperty

我们可以使用下面的方式来定义属性：

这样做也能够达到目的。但是Object.defineProperty允许我们用属性描述器来定义我们的属性。我们可以定义属性是否可写、可枚举或可配置。通常，我们并不需要这么复杂的方式来赋值属性。但这里使用Object.defineProperty有个好处：用属性描述器定义的属性默认是只读的。这可以防止一些经验不高的开发者犯些低级错误：

使用Object.defineProperty能够保护引入的库不被重新赋值。如果尝试重写，程序会抛出“TypeError: Cannot assign to read only property”的错误。

$

$符号，我们用$开头的属性存放引入的库。这种做法不是强制的，只是为了提醒这些属性是公有的。你可以在任何地方使用。反之，某些属性只能在Vue内部使用。

this

我们可以使用this.$libraryname来访问引入的库。但是，我们需要保证this指针指向的是Vue实例。通常，在回调函数中直接使用this，this并不指向Vue实例。

当然，可以使用箭头函数来解决回调this指向不对的问题。也可以在回调函数前，定义let self = this，在回调函数中使用self也可以解决这个问题。

语法意义：创建一个Vue实例。

和面向对象语言（如：Java）新建一个对象类似。我们可以看到，new语法后跟了一个大括号，里面有四个语句。

单文件组件——App.vue

我们在main.js里的template可以看到值为<App/>。第一次看到时，可能会疑惑，这是什么呢？从main.js的import语句，我们可以看到import App from './App'。我们查看目录，可以看到目录下有个App.vue文件。

这里，我们需要学习到Vue一个重要的知识点——单文件组件。文件扩展名为.vue是Vue的单文件组件。以下内容是Vue官网对于单文件组件的介绍：

在很多 Vue 项目中，我们使用 Vue.component 来定义全局组件，紧接着用 new Vue({ el: ‘#container ‘}) 在每个页面内指定一个容器元素。
这种方式在很多中小规模的项目中运作的很好，在这些项目里 JavaScript 只被用来加强特定的视图。但当在更复杂的项目中，或者你的前端完全由 JavaScript 驱动的时候，下面这些缺点将变得非常明显：
全局定义 (Global definitions) 强制要求每个 component 中的命名不得重复
字符串模板 (String templates) 缺乏语法高亮，在 HTML 有多行的时候，需要用到丑陋的
不支持 CSS (No CSS support) 意味着当 HTML 和 JavaScript 组件化时，CSS 明显被遗漏
没有构建步骤 (No build step) 限制只能使用 HTML 和 ES5 JavaScript, 而不能使用预处理器，如 Pug (formerly Jade) 和 Babel
文件扩展名为 .vue 的 single-file components(单文件组件) 为以上所有问题提供了解决方法，并且还可以使用 Webpack 或 Browserify 等构建工具。

3. package.json

这是一个JSON配置文件，管理你项目开发的依赖。如果你是一个Java开发者，那么你肯定用过maven或gradle管理项目的依赖。这个package.json文件也是如此。

package.json必须包含两个字段：“name”、“version”.。还有很多其他的字段，大家可以自行查阅相关文档。

在项目中，规定此项目的依赖包，在文件中可以定义为两种类型：“dependencies”、“devDependencies”。前者是在生成环境中需要依赖的包，后者仅在开发和测试环境需要的包。

大家可以通过手动在package.json文件中添加，但更佳的方式是使用npm install的–save和–save-dev命令。前者将依赖写入到“dependencies”中，后者写入到“devDependencies”。

这样，其他人使用你的项目，只需要执行命令npm install即可。

4. webpack

webpack是现代JavaScript开发的模块打包工具。webpack中文网站大家更详细的信息可以访问这个网址学习webpack。

这里简要记录webpack的四个核心概念：

  • 入口（Entry）：告诉webpack应该从哪个文件开始
  • 出口（Output）：将所有资源获取到之后，告诉webpack在哪里打包应用。
  • Loader：webpack自身只理解JavaScript，并且webpack将每个文件（.css,.html,.scss,.jpg,etc）都作为模块处理，Loader就是将这些转换为模块。
  • 插件（Plugins）：由于 loader 仅在每个文件的基础上执行转换，而 插件(plugins) 更常用于（但不限于）在打包模块的 “compilation” 和 “chunk” 生命周期执行操作和自定义功能。

design-pattern

原文：https://quanke.gitbooks.io/design-pattern-java

设计模式学习记录1

在工作开发的过程中，总是能看到很多地方用到了设计模式。好的设计模式能够让我们事半功倍。前人的基础上总结了许多好的设计模式，现在系统的学习并记录。

面向对象的设计模式原则

单一职责

这个很好理解。控制每个类的粒度，每个类只做该做的，便于提高代码的复用性。

单一职责原则，目的是实现高内聚、低耦合。

学会给类减负。

开闭原则

开闭原则由Bertrand Meyer于1988年提出，其定义如下：
开闭原则(Open-Closed Principle, OCP)：一个软件实体应当对扩展开放，对修改关闭。即软件实体应尽量在不修改原有代码的情况下进行扩展。

开闭原则，通俗的说就是对扩展开放，对修改关闭。

在Java的世界里，扩展即可通过接口或抽象类。需要对事物高度抽象化。这样，如果需要修改系统的行为，就无需对抽象层做任何改动，只需要增加新的具体类来实现新的业务功能即可。

例如：一个接口名为水果，具有一个抽象方法吃。现在系统有继承该接口的两个类，苹果和香蕉。当我们想吃其他水果的时候，只需要继承水果这个接口。如，想吃桃，实现名为桃的类，继承该接口即可。

里氏代换

里氏代换原则由2008年图灵奖得主、美国第一位计算机科学女博士Barbara Liskov教授和卡内基·梅隆大学Jeannette Wing教授于1994年提出。其严格表述如下：如果对每一个类型为S的对象o1，都有类型为T的对象o2，使得以T定义的所有程序P在所有的对象o1代换o2时，程序P的行为没有变化，那么类型S是类型T的子类型。这个定义很拗口且难以理解，通俗定义： 里氏代换原则(Liskov Substitution Principle, LSP)：所有引用基类（父类）的地方必须能透明地使用其子类的对象。

里氏代换原则需要注意：

子类中的方法应该都在父类中有声明。否则，该方法无法在以父类定义的对象使用。

设计上，一般将父类设计为基类或接口。

Java在编译时，会检查一个程序是否符合里氏代换原则，但检查是有局限的。

依赖倒转

如果说开闭原则是面向对象设计的目标的话，那么依赖倒转原则就是面向对象设计的主要实现机制之一，它是系统抽象化的具体实现。依赖倒转原则是Robert C. Martin在1996年为“C++Reporter”所写的专栏Engineering Notebook的第三篇，后来加入到他在2002年出版的经典著作“Agile Software Development, Principles, Patterns, and Practices”一书中。依赖倒转原则定义如下： 依赖倒转原则(Dependency Inversion Principle, DIP)：抽象不应该依赖于细节，细节应当依赖于抽象。换言之，要针对接口编程，而不是针对实现编程。

针对抽象类或接口编程，将具体的子类配置在配置文件中。

在大多数情况下，开闭原则、里氏代换原则和依赖倒转原则会同时出现，开闭原则是目标，里氏代换原则是基础，依赖倒转原则是手段，它们相辅相成，相互补充，目标一致，只是分析问题时所站角度不同而已。

给iPhone定制铃声

使用iPhone，想着给自己电话制作需要的铃声，现将步骤记录如下（附图的话，之后再补）：

  1. 下载好需要制作铃声的歌曲
  2. 打开iTunes，将下载好的歌曲添加到iTunes的歌曲中

  3. 选择歌曲，右键打开显示简介，并切换到选项

  4. 在开始和结束设定时间，不要超过40秒

  5. 设置完成后点击确定（MAC下为 好）按钮即可。

  6. Mac下，选择打开文件菜单，找到转换，选择创建ACC版本

  7. 这时就可以看到，生成了一个同名的但是时间更短的文件。

  8. 点击生成的这个文件，在Finder中显示（windows下应为在Windows资源器中显示）

  9. 将文件拓展名M4A修改为M4R

  10. 回到iTunes，右键选择从资源库删除这个文件，但不删除实际的文件

  11. 双击铃声文件，用iTunes打开，这个文件就默认在铃声里了

  12. 连接上iPhone，打开iPhone的界面。

  13. 点击同步或者选择铃声同步即可。注意每一次同步都会把以前同步过的铃声清楚。
